# MovieFlow Constitution - 视频创作核心原则

**规范驱动视频创作的根本法则**

## 核心哲学

视频不为制作服务——制作为规范服务。创意规范不是制作的指南；它是生成制作的源头。分镜脚本不是指导拍摄的文档；它们是生成素材的精确定义。这是对驱动视频制作的根本性重新思考。

---

## I. 成本优先思维 (Cost-First Thinking)

### 原则

每个创意决策都必须考虑成本优化。渐进式验证是唯一路径，从最低成本开始，随着信心增长逐步提升投入。

### 实践

**渐进式验证路径**：
```
L0 验证（免费）→ L1 验证（3-18元）→ L2 验证（28元，可选）→ L3 制作（170元）
```

**具体要求**：
- L0文本分析必须通过后才能进入L1
- L1优先使用wireframe(3元)或主设计图，满意后才full render(6元)
- L2仅在以下情况执行：
  - 复杂相机运动（推拉摇移跟）
  - 角色动作复杂（战斗、舞蹈、跑步）
  - 需要验证转场效果
  - 首次使用新平台
- 从便宜开始，信心增长后才升级

**成本估算透明化**：
- 每个Phase的成本必须在plan.md中明确标注
- 总成本必须在预算内
- 超预算必须有文档化的充分理由

### 质量关口 (Gates)

```markdown
- [ ] L0验证通过（提示词质量≥60分）
- [ ] 总成本已估算并批准
- [ ] 每阶段成本有明确理由
- [ ] 降级策略已文档化
- [ ] 成本/质量权衡已讨论
```

**违反记录**：如果必须跳过某个验证级别，必须在plan.md的Complexity Tracking中记录原因。

---

## II. 规范优先 (Spec-First)

### 原则

规范不为视频服务——视频为规范服务。不可协商：所有创意工作必须从规范开始。

### 实践

**强制流程**：
```
/specify → /plan → /script → /validate → /implement
不得跳过或颠倒顺序
```

**规范完整性**：
- spec.md必须在/implement前完成
- spec.md不得包含实现细节（平台选择在plan.md中）
- spec.md必须可被非技术人员理解
- 所有创意决策必须在spec中体现

**维护即演进**：
- 修改视频 = 修改规范 + 重新生成
- 视频是规范的视觉表达，不是主资产
- 规范版本化，视频可重新生成

### 质量关口 (Gates)

```markdown
- [ ] spec.md已完成并审核
- [ ] L0验证通过
- [ ] 所有[NEEDS CLARIFICATION]已解决
- [ ] 成功标准可衡量
- [ ] 平台需求已明确
```

**违反后果**：在spec.md完成前执行/implement将被终止。

---

## III. 质量通过迭代 (Quality Through Iteration)

### 原则

快速迭代胜过昂贵完美。优化快速反馈循环，而非追求一次性完美。

### 实践

**快速反馈策略**：
- 最初用wireframe/主设计图，而非直接full render
- 渐进式精炼：wireframe → full → 动态 → 最终
- 规范演进：更新规范，而不仅仅是视频

**迭代路径**：
```
Iteration 0: spec.md + plan.md（纯文本）
Iteration 1: L0验证 + wireframe（~3元）
Iteration 2: L0验证 + full render（~6元）
Iteration 3: L2动态预览（~28元，optional）
Iteration 4: L3最终制作（~170元）
```

**快速实验**：
- 支持"假设"实验：如果需要将基调从严肃改为幽默？
- 用更新的规范在20分钟内重新生成
- 而不是重拍

### 质量关口 (Gates)

```markdown
- [ ] 每次迭代有明确的学习目标
- [ ] 反馈周期<30分钟（L0-L1）
- [ ] 规范随学习而演进
- [ ] 迭代成本递增合理
```

## IV. 角色一致性 (Character Consistency)

### 原则

跨场景、跨Episode的角色外貌和风格必须保持一致。角色不一致是AI视频的头号质量问题。

### 实践

**角色素材库系统**：
- 所有主要角色必须建立素材库
- 素材库包含：
  - 13个标准角度参考图（13图融合法）
  - 详细外貌描述（年龄、性别、特征、服装）
  - 显著特征列表（用于提示词固定）
- 维护`character-asset-library.json`

**一致性策略**：
| 平台 | 策略 | 成本 |
|-----|------|------|
| Sora2 | 提示词固定 + lip-sync | 低 |
| Runway | 参考图 + ControlNet | 中 |
| Midjourney | 13图融合 → 统一seed | 高但最可靠 |

**首次出现流程**：
```
新角色首次出现 → 生成13个角度参考图 → 融合 → 入库 → 后续场景复用
```

### 质量关口 (Gates)

```markdown
- [ ] 主要角色已入素材库
- [ ] 跨场景使用统一描述
- [ ] 显著特征在所有场景保持
- [ ] 服装/发型/配饰一致性检查通过
```

**验证标准**：
- 人工审核：角色外貌相似度≥85%
- AI自动检查：特征关键词匹配率100%

## V. 视觉连贯性 (Visual Continuity)

### 原则

全局视觉风格必须统一。色彩调性、光照风格、镜头语言要保持一致性。

### 实践

**全局风格定义**：
在spec.md中明确：
- **色彩调性**：暖色调/冷色调/去饱和/高对比
- **光照风格**：自然光/影棚光/戏剧性光照/柔和光
- **镜头语言**：稳定/运动/手持/电影感
- **美术风格**：写实/动画/Q版/3D渲染

**连贯性追踪**：
维护`visual-continuity.json`：
```json
{
  "globalStyle": {
    "colorGrading": "暖色调，电影感",
    "lighting": "顶光+侧光，柔和自然",
    "cameraStyle": "稳定镜头为主，关键时刻运动"
  },
  "episodeStyles": [
    {
      "episodeId": "EP001",
      "timeOfDay": "黄昏",
      "weather": "晴朗",
      "emotionalTone": "温馨"
    }
  ]
}
```

**跨场景规则**：
- 同一地点的场景：光照/配色/氛围必须一致
- 时间流逝的场景：光照变化要符合逻辑（早→中→晚）
- 情绪转折的场景：色调可以变化，但要流畅过渡

### 质量关口 (Gates)

```markdown
- [ ] 全局风格已在spec中定义
- [ ] visual-continuity.json已创建
- [ ] 跨场景色彩调性一致性≥90%
- [ ] 光照风格符合时间/地点逻辑
- [ ] 转场流畅自然
```

## VI. 平台适配 (Platform Adaptation)

### 原则

不同平台有不同的优势和限制。提示词风格必须适配平台特性，不能一刀切。

### 实践

**平台选择策略**：
```markdown
| 平台 | 适用场景 | 提示词风格 | 优势 | 成本 |
|-----|---------|-----------|------|------|
| Sora2 | 写实、lip-sync | ai-optimized | 物理准确、唇形同步 | 高 |
| Runway Gen-4 | 简洁场景、相机控制 | 简洁描述+数值化 | 相机控制强 | 中 |
| 即梦AI v3.0 | 中文内容、首尾帧 | 详细中文 | 成熟稳定 | 中 |
| 海螺02 | 快速迭代 | 标准描述 | 速度快 | 低 |
| 可灵 | 预算紧张 | 简单描述+运动控制 | 成本低 | 低 |
```

**提示词翻译规则**：
- **Sora2** (ai-optimized)：
  ```
  "Character says 'text' at 0:06, lip-synced, [emotion]"
  "Heavy object (3kg), arms bent from weight"
  "Music: Jazz, BPM 95, ducks to -28dB during dialogue"
  ```

- **Runway** (简洁+数值)：
  ```
  "简洁视觉描述"
  Camera: Horizontal=+3, Vertical=-2, Zoom=+5
  ```

- **即梦AI** (详细中文+首尾帧)：
  ```
  首帧: 女性站立，面向镜头
  尾帧: 女性转身背对镜头
  描述: 女性缓慢转身，从正面变为背影
  ```

**降级策略**：
在plan.md中定义：
```markdown
| 失败场景 | 切换到 | 原因 |
|---------|--------|------|
| 角色一致性差 | Sora2 | lip-sync能力强 |
| 预算超支 | 可灵 | 成本较低 |
| 生成超时 | 海螺02 | 速度快 |
```

### 质量关口 (Gates)

```markdown
- [ ] 平台选择有明确理由（记录在plan.md）
- [ ] 提示词风格适配平台特性
- [ ] 降级策略已文档化
- [ ] 跨平台一致性验证通过
```

## VII. 三层画面结构 (Three-Layer Composition)

### 原则

9:16竖屏的黄金法则：每个镜头必须有前景、中景、背景三层。这是AI生成视频避免"平面感"的关键。

### 实践

**标准结构**：
```
前景 (20%): 景深元素 / 动态元素 / 框架元素
中景 (60%): 主体内容 / 角色 / 关键动作
背景 (20%): 环境设置 / 氛围营造 / 空间感
```

**具体示例**：
```markdown
| 镜头类型 | 前景 | 中景 | 背景 |
|---------|------|------|------|
| 对话场景 | 另一角色的肩膀 | 说话的角色 | 办公室环境 |
| 动作场景 | 飞扬的尘土 | 打斗的角色 | 破败的院落 |
| 环境展示 | 飘落的树叶 | 建筑主体 | 远山和天空 |
```

**在提示词中体现**：
```
画面层次：
前景 - [具体元素，如飘落的叶子]
中景 - [主体内容，如主角站立]
背景 - [环境设置，如古代建筑]
```

### 质量关口 (Gates)

```markdown
- [ ] 每个镜头都有三层结构
- [ ] 前景有视觉趣味（不是空白）
- [ ] 中景主体清晰（占据60%视觉权重）
- [ ] 背景营造氛围（而非干扰）
```

**例外情况**：
- 特写镜头可以背景虚化，但前景仍需保留
- 纯黑/纯白背景需要在spec中特别说明

## VIII. 10秒情绪单元 (10-Second Emotional Unit)

### 原则

每个10秒场景是一个完整的情绪单元，有明确的情绪起点和终点。这是短视频叙事的基本规律。

### 实践

**情绪弧线设计**：
```
起点（0-2秒）: 建立情绪基调
发展（3-7秒）: 情绪推进/冲突展开
高潮（8-9秒）: 情绪峰值
收尾（9-10秒）: 情绪落地/引导下一场景
```

**6场景情绪规划**：
```
场景1: 平静 → 好奇
场景2: 好奇 → 惊讶
场景3: 惊讶 → 紧张
场景4: 紧张 → 冲突
场景5: 冲突 → 高潮
场景6: 高潮 → 释然
```

**独立性与连贯性平衡**：
- **可独立理解**：每个10秒看懂在讲什么
- **连接流畅**：场景1的结束自然引导场景2的开始
- **整体叙事**：6个场景组成完整的情绪旅程

### 质量关口 (Gates)

```markdown
- [ ] 每个场景有明确的情绪起点和终点
- [ ] 10秒内完成一个完整的情绪弧线
- [ ] 场景间情绪过渡流畅
- [ ] 整体符合三幕结构或Story Beats
```

## IX. 多阶段工作流 (Multi-Phase Workflow)

### 原则

复杂项目可以通过多阶段工作流提升质量。阶段不是命令，而是规范中定义的Phase。

### 实践

**标准阶段定义**（在plan.md中选择）：

```markdown
### Phase 0: 主设计图生成 (Optional)
**Purpose**: 为每个场景生成关键帧参考图
**Platform**: Midjourney V6 / FLUX / Stable Diffusion
**Output**: 6张关键帧设计图
**Cost**: ~18元 (6张 × 3元)
**When to Use**:
- 需要精确视觉控制
- 角色/场景复杂
- 需要客户预先确认

### Phase 1: 置景设计 (Optional)
**Purpose**: 3D场景布局和环境设计
**Platform**: Blender / Unreal / 3D工具
**Output**: 场景布局文件/参考图
**Cost**: ~[自定义]
**When to Use**:
- 需要精确空间关系
- 多角度拍摄同一场景

### Phase 2: 打光设计 (Optional)
**Purpose**: 专业光照方案设计
**Platform**: 光照工具 / 手动标注
**Output**: 光照参数/参考图
**Cost**: ~[自定义]
**When to Use**:
- 需要电影级光照
- 特殊光影效果

### Phase 3: 视频生成 (Mandatory)
**Purpose**: 生成6个10秒视频片段
**Platform**: Sora2 / Runway / 即梦AI / 其他
**Output**: 6个视频文件
**Cost**: ~170元
**Always Required**: 这是核心阶段

### Phase 4: 4K放大 (Optional)
**Purpose**: 成片分辨率增强/细节优化
**Platform**: Topaz Video AI / 其他放大工具
**Output**: 4K视频
**Cost**: ~[自定义]
**When to Use**:
- 需要更高分辨率
- 细节不足需增强
```

**阶段编排逻辑**：
```
if (spec要求主设计图) → 执行Phase 0
if (spec要求置景) → 执行Phase 1
if (spec要求打光) → 执行Phase 2
始终执行Phase 3
if (spec要求4K放大) → 执行Phase 4

每个Phase的输出作为下一个Phase的输入
```

### 质量关口 (Gates)

```markdown
- [ ] 在spec.md中明确勾选需要的阶段
- [ ] 在plan.md中详细定义每个Phase
- [ ] Phase间数据传递路径清晰
- [ ] 总成本在预算内
```

## X. 宪法遵守 (Constitution Compliance)

### 原则

宪法是MovieFlow视频创作的根本法则。所有决策必须符合宪法，违反必须有充分理由。

### 实践

**Constitution Check在plan.md中**：
```markdown
## Constitution Check
*GATE: 必须在Phase 0前通过，Phase 1后重新检查*

### I. 成本优先思维
- [ ] 渐进式验证路径已规划
- [ ] L0→L1→(L2?)→L3成本递增合理
- [ ] 总成本在预算内

### II. 规范优先
- [ ] spec.md已完成
- [ ] L0验证已通过
- [ ] 所有NEEDS CLARIFICATION已解决

### III. 质量通过迭代
- [ ] 迭代路径清晰
- [ ] 反馈周期合理

### IV-VIII. (其他原则)
...
```

**Complexity Tracking**（记录违反）：
```markdown
## Complexity Tracking
*仅在Constitution Check有违反时填写*

| 违反原则 | 为什么必需 | 更简单方案被拒绝的原因 |
|---------|----------|---------------------|
| 跳过L1验证 | 预算极度紧张 | 风险可接受，历史成功率高 |
| 不使用三层结构 | 特写镜头为主 | 故事需要，背景干扰情绪 |
```

**审核流程**：
1. 在/plan时自动运行Constitution Check
2. 如有违反，必须填写Complexity Tracking
3. 如无法合理解释，/plan失败，要求修改spec
4. 在Phase 1后重新检查，防止设计偏离

### 质量关口 (Gates)

```markdown
- [ ] Initial Constitution Check: PASS
- [ ] Post-Design Constitution Check: PASS
- [ ] 所有违反有文档化的理由
- [ ] Complexity Tracking已填写完整
```

## Governance

### 宪法优先级

1. **宪法优先于一切**：宪法 > 最佳实践 > 个人偏好
2. **修订流程**：修改宪法需要：
   - 文档化的理由
   - 广泛讨论
   - 版本更新
3. **导演最终决策权**：导演可以违反宪法，但必须：
   - 在Complexity Tracking中记录
   - 解释为什么必要
   - 承担责任

### 执行机制

- 所有/plan必须通过Constitution Check
- AI助手有权拒绝不符合宪法的请求
- 违反宪法的视频生成会在QA阶段被标记
- 宪法遵守度是质量评分的重要指标

### 持续改进

- 定期回顾宪法的实际执行情况
- 收集违反案例，分析是否需要修订
- 版本化管理，记录修订历史
